# GoalX Lightweight Goal Boundary Design

## Status

Drafted on 2026-03-24 after local review plus parallel subagent review.

## Decision

Replace the current `goal-contract + acceptance + proof` protocol with a lighter goal-boundary model:

- immutable user objective anchor in `run-spec.yaml` / `run-metadata.json`
- mutable `goal.json` as the current boundary snapshot
- mutable `acceptance.json` as the current verification gate
- append-only `goal-log.jsonl` as audit-only boundary/path-change history
- canonical `proof/completion.json` generated by `goalx verify`

This is a hard cutover. Do not keep dual writable semantics for `goal-contract.json`.

## Problem

The current model is over-coupled in three ways:

1. `goal-contract.json` mixes boundary, execution, and proof semantics.
2. `acceptance.json` mixes gate definition, override policy, and latest result.
3. `proof/completion.json` is mostly a mirror of the other two, so proof is not truly canonical.

That over-coupling creates two product failures:

- master spends too much effort maintaining protocol paperwork instead of exploring and pushing the goal forward
- the framework rewards self-grading fields (`done`, `semantic_match`, `counter_evidence`) instead of enforcing a small number of hard closeout rules

## Goals

- Keep the user's original goal immutable.
- Let the master evolve the current boundary and switch paths autonomously.
- Make exploration and path switching a default behavior for non-trivial goals.
- Keep boundary state small, readable, and auditable.
- Keep execution ownership and liveness out of the boundary schema.
- Make `goalx verify` the only canonical closeout writer.
- Remove `goal-contract.json` as an active protocol file.

## Non-Goals

- Do not turn GoalX into a full event-sourced system.
- Do not keep long-term compatibility write paths for old boundary files.
- Do not require the master to maintain rich proof metadata on every control cycle.
- Do not allow the master to silently narrow or waive user-required outcomes.

## Model Summary

### 1. Immutable User Objective

The immutable user objective remains in:

- `run-spec.yaml`
- `run-metadata.json`

This is the anchor. The master may reinterpret or expand the path to success, but it may not rewrite the original user ask.

### 2. `goal.json`

`goal.json` is the current boundary snapshot. It answers:

- what outcomes are currently required for this run or phase
- what optional improvements are worth pursuing
- which items the master currently believes are ready for verification
- which items were explicitly waived by the user

It does not answer:

- who owns the work
- whether work is blocked
- whether a session is alive
- which path is currently dispatched

Proposed shape:

```json
{
  "version": 1,
  "updated_at": "RFC3339",
  "required": [
    {
      "id": "req-1",
      "text": "build the knowledge base from primary sources",
      "source": "user|master",
      "state": "open|claimed|waived",
      "evidence_paths": ["/abs/path"],
      "note": "short claim summary",
      "user_approved": false
    }
  ],
  "optional": [
    {
      "id": "opt-1",
      "text": "add searchable summaries",
      "source": "master",
      "state": "open|claimed|waived",
      "evidence_paths": [],
      "note": "",
      "user_approved": false
    }
  ]
}
```

Rules:

- `open` means still required or optional and not yet ready for verification.
- `claimed` means the master believes the item is satisfied and has concrete evidence paths ready for `verify`.
- `waived` only counts for required items when `user_approved=true`.
- `source=user` items are user-locked. They may be clarified or split, but not dropped or narrowed without user approval.
- `source=master` items may be added when necessary to reach the objective.
- `goal.json` is the only mutable goal-boundary source of truth.

### 3. `acceptance.json`

`acceptance.json` is the current verification gate, not a checklist contract.

Proposed shape:

```json
{
  "version": 1,
  "goal_version": 3,
  "default_command": "go test ./...",
  "effective_command": "go test ./... && ./smoke.sh",
  "change_kind": "same|expanded|rewritten|narrowed",
  "change_reason": "why the current gate is the right gate",
  "user_approved": false,
  "last_result": {
    "status": "pending|passed|failed",
    "checked_at": "RFC3339",
    "exit_code": 0,
    "evidence_path": "/abs/path"
  }
}
```

Rules:

- `effective_command == default_command` is the normal case.
- if the gate changes, `change_kind`, `change_reason`, and `goal_version` must be updated together.
- `narrowed` requires explicit `user_approved=true`
- `expanded` and `rewritten` may be chosen autonomously when they increase verification quality and do not weaken user scope.
- `acceptance.json` is the only mutable gate source of truth.

### 4. `goal-log.jsonl`

`goal-log.jsonl` is audit-only. It is not a read-model and not a state source.

Allowed event types:

- `path_compared`
- `path_selected`
- `path_switched`
- `goal_expanded`
- `goal_split`
- `acceptance_changed`
- `waiver_requested`
- `waiver_approved`

This file exists to preserve reasoning around boundary changes and path switches without turning GoalX into a full event-sourced system.

### 5. `proof/completion.json`

`goalx verify` generates the only canonical closeout artifact.

Proposed shape:

```json
{
  "version": 1,
  "goal_version": 3,
  "acceptance_status": "passed|failed|pending",
  "goal_satisfied": true,
  "required_total": 3,
  "required_satisfied": 3,
  "required_remaining": 0,
  "completion_mode": "verification_only|implementation_and_verification",
  "code_changed": false,
  "base_revision": "abc123",
  "head_revision": "def456",
  "changed_files": ["a.go"],
  "items": [
    {
      "goal_item_id": "req-1",
      "verdict": "satisfied|waived|unsatisfied",
      "basis": "preexisting|run_change|mixed|waived",
      "evidence_paths": ["/abs/path"],
      "note": "short proof note",
      "user_approved": false
    }
  ],
  "updated_at": "RFC3339"
}
```

Rules:

- this file is written by `goalx verify`, not by the master during normal orchestration
- `goal_satisfied=true` requires:
  - current `goal.json` version matches `goal_version`
  - all required items are `satisfied` or `waived`
  - any waived required item has `user_approved=true`
  - `acceptance_status=passed`
- optional items may appear, but they do not block closeout

## Behavior Rules

### Master Autonomy

For any non-trivial goal, unclear root cause, or medium+ capability slice:

1. the master must compare at least 2-3 paths before committing
2. the comparison must include, at minimum:
   - local patch path
   - compatibility-preserving path
   - architecture-level path
3. the master must choose one path and record a short reason
4. if later evidence shows the chosen path is stuck, falsified, or clearly dominated, the master must switch paths autonomously

This is default behavior. It does not require an extra user `tell`.

### Master Boundary Authority

The master may:

- add `source=master` required outcomes when truly necessary
- add or remove optional outcomes
- rewrite or expand the effective acceptance gate
- split one user outcome into smaller user-locked outcomes
- switch paths and dispatch sessions accordingly

The master may not:

- delete or narrow `source=user` required outcomes without user approval
- mark a required item `waived` without user approval
- narrow the effective acceptance gate without user approval

### Execution Ownership

Execution semantics stay outside the boundary model:

- inbox
- lease state
- runtime state
- coordination digest
- journals

Those files track who is working, who is blocked, and what to do next. They do not redefine the goal.

## Verification Rules

`goalx verify` must do four things:

1. run the current effective acceptance command
2. validate that `goal.json` is non-empty on required scope
3. validate every `claimed` required item has concrete evidence paths
4. produce canonical `proof/completion.json`

Hard rules:

- a required item in `open` state is unsatisfied
- a required item in `claimed` state is only satisfied if verify can accept its evidence
- a required item in `waived` state is only satisfied if `user_approved=true`
- if `goal.json` changes, prior acceptance results and completion proofs become stale immediately

## Read Models

These remain derived views:

- `state/run.json`
- `state/sessions.json`
- project `status.json`
- `goalx status`
- `goalx observe`

They may expose fields like:

- `goal_required_total`
- `goal_required_claimed`
- `goal_required_verified`
- `goal_required_remaining`
- `acceptance_status`
- `completion_mode`

But those are summaries, not protocol truth.

## Cutover Rules

- Hard cutover only. No dual-write between `goal-contract.json` and `goal.json`.
- `goal-contract.json` is removed from active protocol and template vocabulary.
- `acceptance.md` is demoted to optional notes or removed from startup.
- `proof/completion.json` keeps its path but gets a simplified schema.
- old active runs must be treated as pre-cutover artifacts, not silently rewritten in place.

## Why This Is Better

- The user objective stays anchored.
- The master can evolve the path and boundary without rewriting the user ask.
- Exploration and path switching become default operating behavior.
- Boundary state becomes small enough that the master can maintain it without paperwork dominating the run.
- Verify becomes the only closeout writer, so completion authority is clear.
